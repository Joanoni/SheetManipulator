## Root Cause Analysis

<thinking>

### Primary Root Cause ‚Äî Vite Dependency Pre-Bundle Cache Baked Into Docker Image

Vite's dev server pre-bundles dependencies and caches transformed source files in `.vite/` (inside the project's `node_modules/.vite/` or the configured `cacheDir`). When `npm run dev` starts inside the Docker container, Vite checks this cache directory.

**The problem:** The `src/frontend/Dockerfile` copies the entire project directory into the image (`COPY . .`) **after** running `npm ci`. If the developer's local machine has a `.vite/` cache directory (generated by a previous local `npm run dev` run) that predates the T-008 addition of `ColumnDefinition` to `schema.ts`, that stale cache directory gets baked into the Docker image layer. When the container starts, Vite finds the cache, considers it valid (the file hash may not have changed from its perspective inside the container), and serves the stale pre-transformed module ‚Äî which does not contain the `ColumnDefinition` export.

The browser then receives the stale ESM module and throws the `SyntaxError` because the named export is absent from the cached transform.

### Secondary Issue ‚Äî `VITE_API_URL=http://localhost:8000` Is Incorrect for Docker

In `src/docker-compose.yml` line 19:
```yaml
environment:
  - VITE_API_URL=http://localhost:8000
```

`VITE_*` environment variables are **inlined at build time** by Vite, not at runtime. Since the Dockerfile uses `CMD ["npm", "run", "dev", "--", "--host"]` (dev server, not a production build), the variable is read at dev-server startup. However, `http://localhost:8000` resolves to the **user's browser machine**, not the backend container. This works coincidentally when the user runs Docker on their local machine (because the backend port 8000 is published to `localhost:8000` on the host), but it is architecturally incorrect and will break in any remote or CI environment.

This is a pre-existing issue (present before R-001) and is lower severity than the cache issue, but should be documented.

### Primary Root Cause ‚Äî Stale Vite Cache Baked Into Docker Image Layer

**Mechanism:**

1. Developer runs `npm run dev` locally at some point before T-008 was implemented. Vite creates `.vite/` cache in `node_modules/.vite/` containing pre-transformed modules ‚Äî including a version of `schema.ts` **without** `ColumnDefinition`.
2. Developer runs `docker compose up --build`. The `Dockerfile` executes:
   ```dockerfile
   COPY package.json package-lock.json ./
   RUN npm ci                          # installs node_modules/ fresh
   COPY . .                            # ‚Üê copies local node_modules/.vite/ cache into image
   ```
3. The stale `.vite/` cache directory is now baked into the Docker image layer.
4. Container starts, Vite dev server finds the cache, considers it valid, and serves the stale pre-transformed `schema.ts` ‚Äî which does not contain `ColumnDefinition`.
5. Browser receives the stale ESM module and throws `SyntaxError`.

**Evidence:** The export exists in the source file. The error only manifests in Docker (not local dev). The R-001 fix removed the bind mount, so the container now uses the image-baked files exclusively ‚Äî making the stale cache the only possible source of the discrepancy.

### Secondary Issue ‚Äî `VITE_API_URL` Architectural Concern

`VITE_API_URL=http://localhost:8000` in `docker-compose.yml` is read by the Vite dev server at startup and inlined into the bundle. This works on a local machine (backend port is published to host `localhost:8000`) but is architecturally incorrect ‚Äî it will fail in remote/CI environments. Lower severity; documented for completeness.

## Fix Recommendations

### Fix A ‚Äî Exclude `.vite/` Cache from Docker Build Context (Recommended)

Add a `.dockerignore` file in `src/frontend/` to prevent the local Vite cache from being copied into the image:

**`src/frontend/.dockerignore`:**
```
node_modules/
.vite/
dist/
```

This is the minimal, correct fix. The `node_modules/` exclusion also prevents the local `node_modules/` from overwriting the `npm ci`-installed one (defense-in-depth, even though R-001 removed the bind mount).

### Fix B ‚Äî Force Vite Cache Invalidation on Container Start (Alternative)

Modify `src/frontend/Dockerfile` CMD to pass `--force` to Vite:

```dockerfile
CMD ["npm", "run", "dev", "--", "--host", "--force"]
```

`--force` causes Vite to ignore and regenerate the dependency pre-bundle cache on every startup. This is less efficient (slower cold start) but guarantees cache freshness.

### Fix C ‚Äî Switch to Production Build + Static Server (Long-term)

Replace the dev server with a production build in the Dockerfile:

```dockerfile
FROM node:20-alpine AS builder
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:20-alpine
WORKDIR /app
RUN npm install -g serve
COPY --from=builder /app/dist ./dist
EXPOSE 5173
CMD ["serve", "-s", "dist", "-l", "5173"]
```

This eliminates the Vite dev server entirely from Docker, removes the cache problem at its root, and produces a smaller, faster container. Recommended for production readiness.

## Recommended Action

**Implement Fix A** (`src/frontend/.dockerignore`) as the immediate fix ‚Äî it is a one-line change that resolves the root cause without altering the Dockerfile or dev workflow. **Also implement Fix B** (`--force` flag) as a safety net. Document Fix C as a future improvement.
